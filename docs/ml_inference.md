# ML Inference — How the Trend-Strength Score Is Produced

This document describes the current ML scoring pipeline end-to-end: what data it trains on, how the label is constructed, what the model learns, and how inference runs on every brief. For the training commands, see [`docs/commands.md`](./commands.md). For the roadmap, see [`ml_roadmap.md`](./ml_roadmap.md).

---

## Core idea

Every time the pipeline runs, it takes a snapshot of four market dimensions (HTF structure, derivatives positioning, ETF flows, exchange flows). Each snapshot is a vector of ~77 numbers describing the current state of the market.

The ML model takes that vector and answers one question:

> **"Given what the market looks like right now, will price be higher or lower in 168 hours?"**

The answer is a number in **[-1, +1]**:
- `+1` → model expects strong upward move over the next week
- `-1` → model expects strong downward move
- `0`  → no clear edge, market is ranging

This number becomes `confluenceTotal`, which determines direction (LONG/SHORT), and position size (larger conviction = larger size).

---

## Why train on snapshots, not trade ideas

The previous approach trained on trade idea outcomes: the pipeline generated a trade idea, and later checked whether it worked. That had three fundamental problems:

1. **Selection bias** — trade ideas were only generated when the heuristic was confident. The model learned only from states the heuristic already approved, inheriting its blindspots.
2. **Look-ahead label** — the training label used the peak `qualityAtPoint` across all 720 recorded hours. Picking the best exit point in hindsight is not a realistic trading label.
3. **Too little data** — ~300 trade ideas over 2 months is not enough to train anything meaningful.

The snapshot model fixes all three:

| Problem | Fix |
|---|---|
| Selection bias | Train on **every snapshot**, not just heuristic-approved ones |
| Look-ahead label | Use the actual forward return at a **fixed horizon** (168h) |
| Too little data | 600 snapshots × 4 horizons = **2,000+ training rows** and growing |

## Architecture

```
Every brief:

Raw context (per dimension)
    │
    ▼
[extract-features.ts]
  → RawFeaturesByDim: 77 numbers across 4 dims

    │
    ├── [snapshot-ml.ts]          ← PRIMARY: single cross-dim model
    │   snapshot_{asset}_v1.onnx    trained on price data (no trade ideas)
    │       │ model missing ↓
    ├── [intradim-ml.ts] L2a      ← FALLBACK: per-dim models
    │   [ml-aggregator.ts] L1       trained on qualityAtPoint (legacy)
    │       │ models missing ↓
    └── heuristic equal-weight mean

    ▼
confluenceTotal ∈ [-1, +1]
    ├── direction: sign → LONG / SHORT
    └── size: |confluenceTotal| → position size %
```

The snapshot model is tried first. If the ONNX file is missing or inference fails, the pipeline silently falls back to the L2a + L1 stack, then to the heuristic. A brief is always produced.

## Training data — how it is built

Training data is generated by `generate-snapshot-training-data.ts` (run via `pnpm ml:gen-training-data`).

### Step 1: Anchor on HTF snapshots

For each `HtfSnapshot` row where `snapshotPrice` is recorded, one time point `T` is created. HTF snapshots run every 4H and always record the spot price — this is the price reference point.

### Step 2: Join all four dimensions at time T

For each HTF snapshot at time T, the script finds the most recent row from each of the other three dimension tables at or before T (binary search, sorted by timestamp):

- `derivatives_snapshots` — latest row with `timestamp ≤ T`
- `etfs_snapshots` — latest row with `timestamp ≤ T`
- `exchange_flows_snapshots` — latest row with `timestamp ≤ T`

This produces a complete market snapshot as of time T across all four dimensions.

### Step 3: Extract the feature vector

The four snapshot rows are wrapped into `DimensionOutput` objects and passed to `extractRawFeatures()` — the same function that runs during live inference. The feature extraction is identical: numerics passed through, categoricals encoded via `feature_schema.json`. The result is a `RawFeaturesByDim` with 77 numbers across four dimensions.

### Step 4: Look up the forward price

The script finds the `HtfSnapshot.snapshotPrice` at approximately `T + horizon` (default 168h). It searches within a ±4h tolerance window for the nearest snapshot price.

The forward return:
```
return_pct = (price_at_T+168h - price_at_T) / price_at_T
```

This is the label: positive = price went up, negative = price went down.

### Step 5: Write to CSV

One row per (snapshot × horizon) pair. The CSV has 77 feature columns (named `{DIM}_{key}`) plus `return_pct`, `timestamp`, `asset`, `horizon_hours`, `price_now`, `price_future`.

Multiple horizons (24h, 48h, 72h, 168h) are stored in the same CSV so the training script can pick any. Current default: **168h** (1 week — consistent with the swing-trading timeframe).

## The label

The raw `return_pct` is normalized before training:

```python
y = clip(return_pct / (3 × std(return_pct)), -1, +1)
```

This maps a 3-sigma return to ±1. At BTC's typical weekly volatility (σ ≈ 3.9%), a +11.5% week maps to `y = +1` and a -11.5% week maps to `y = -1`. Moves inside 1σ map to ±0.33.

The model is therefore learning: **"Is this snapshot followed by an unusually strong directional move?"** — not just direction, but also conviction.

## The model

**Architecture**: Ridge regression (L2 regularization).

Ridge was chosen over Lasso or tree-based models because:
- 77 features with ~400 training rows requires heavy regularization
- Many features are correlated (e.g., multiple CVD signals, multiple MFI/RSI variations) — L2 shrinks all of them smoothly instead of zeroing them
- Ridge has a closed-form solution; no convergence issues at any alpha

**Regularization**: `alpha=10`. This is intentionally strong because with 77 features and ~400 rows, L/N ≈ 0.2 — well into the overfit regime without regularization. Strong Ridge ensures the model produces stable direction predictions (high IC) even if the magnitude predictions are shrunk.

**Training procedure**:
1. Load CSV, filter to the chosen horizon (default 168h)
2. Fill missing features with 0.0 (older snapshots may lack newer features)
3. Normalize target: `y = clip(return_pct / (3 × std), -1, 1)`
4. Walk-forward CV: `TimeSeriesSplit` with 5 folds, evaluating R², MAE, Pearson IC, and p-value
5. Train final model on all data
6. Export to ONNX, write `.meta.json`

**First training results (BTC, 168h horizon, 421 samples, 2026-03-26 → 2026-05-21)**:

| Metric | OOF (held-out) | In-sample |
|---|---|---|
| Pearson IC | +0.630 (p<0.001) | +0.951 |
| R² | 0.096 | 0.904 |
| MAE | 0.245 | 0.080 |

The large gap between in-sample and OOF R² is expected — Ridge heavily shrinks coefficient magnitudes, which inflates in-sample fit relative to OOF. The IC staying high (0.630 OOF vs 0.951 in-sample) confirms that the **direction** prediction is real, even if the predicted magnitude is shrunk toward zero.

**Interpretation**: An OOF IC of 0.63 means the model's rank ordering of snapshots by predicted return closely matches the actual rank ordering of realized returns across held-out folds. For reference, factor models with IC > 0.1 are considered actionable in institutional quant finance; 0.63 is exceptionally strong and likely reflects genuine structure in the 2-month window (not necessarily fully generalizable).

**Top predictive features** (from first BTC training):

| Feature | IC with label | Interpretation |
|---|---|---|
| `HTF_structuralPriceVsPocPct` | −0.730 | Stretched price above structural VPOC → tends to revert down |
| `HTF_structuralVpPosition` | +0.704 | Position below value area → bullish gravity toward VPOC |
| `HTF_biasSthGravity` | +0.698 | STH cost basis above current price → bearish pressure resolved |
| `HTF_priceVsSthPct` | −0.660 | Price far above STH cost basis → stretched, tends to compress |
| `HTF_mfiH4` | −0.516 | High 4H MFI → overbought, tends to revert |
| `ETFS_reversalRatio` | −0.464 | ETF flow reversal after a streak → momentum exhaustion |
| `DERIVATIVES_cbPremiumPercentile1m` | +0.448 | Coinbase premium at extreme → strong spot demand |

Note: most top features measure **mean-reversion from stretched price levels**, not pure momentum continuation. This is not contradictory with the momentum thesis — it means the model is capturing "how extended is the current move?" and correctly predicting the partial reversion that follows overshoots.

## Inference — how it runs on every brief

**File**: `packages/pipeline/src/orchestrator/trade-idea/snapshot-ml.ts`

### 1. Model loading

On the first brief after startup, the inference module loads:
- `models/snapshot_{asset}_v1.onnx` — the ONNX graph
- `models/snapshot_{asset}_v1.meta.json` — contains `feature_order` (the canonical 77-element list)

The loaded session is cached in a `Map` keyed by `{asset}_{version}`. Cache is success-only — if the file is missing, the module returns `null` silently and the pipeline falls back to L2a + L1. A freshly trained model is picked up on the next brief without restarting.

### 2. Feature vector assembly

```typescript
const vec = new Float32Array(77);
for (let i = 0; i < featureOrder.length; i++) {
  const [dim, ...rest] = featureOrder[i].split("_");
  const key = rest.join("_");
  vec[i] = rawFeatures[dim]?.[key] ?? 0.0;  // missing features default to 0
}
```

`rawFeatures` is a `RawFeaturesByDim` object already produced by `extractRawFeatures()` at the start of `processTradeIdea()`. No extra work needed — the same object used for the fallback models is reused.

### 3. ONNX inference

```typescript
const tensor = new ort.Tensor("float32", vec, [1, 77]);
const results = await session.run({ X: tensor });
const raw = results["variable"].data[0];        // scalar from Ridge output
const score = Math.max(-1, Math.min(1, raw));   // clamp to [-1, +1]
```

The output node is named `"variable"` (standard for sklearn Ridge in skl2onnx). The raw output can exceed [-1, +1] slightly if the model sees an unusual input; clamping bounds it.

### 4. Return

```typescript
return { score, modelVersion: "snapshot_btc_v1" }
```

`score` becomes `confluenceTotal`. `modelVersion` is stored in the `TradeIdea.confluence` JSONB for audit.

## Pipeline integration

In `processTradeIdea()` (`packages/pipeline/src/orchestrator/trade-idea/index.ts`):

```typescript
const rawFeatures = extractRawFeatures(outputs);

const snapshotResult = await runSnapshotMl(asset, rawFeatures);

if (snapshotResult) {
  // Snapshot model ran — use its score directly
  confluenceTotal = snapshotResult.score;
  ml = { mlTotal: snapshotResult.score, modelVersion: snapshotResult.modelVersion };
  intradimMl = await runIntradimMl(asset, rawFeatures);  // for display only
} else {
  // Fallback: L2a per-dim models → L1 aggregator
  intradimMl = await runIntradimMl(asset, rawFeatures);
  const confluence = { /* per-dim scores */ };
  ml = await runMlAggregator(asset, confluence);
  confluenceTotal = ml?.mlTotal ?? getConfluenceTotal(confluence);
}
```

The per-dim intradim scores are always computed for display in the sidebar breakdown. The snapshot model only controls `confluenceTotal`.

## Retraining workflow

Retrain whenever new snapshots accumulate (weekly cadence is a good default):

```bash
# Step 1: regenerate the training CSV from all stored snapshots
pnpm ml:gen-training-data             # BTC
pnpm ml:gen-training-data --asset ETH

# Step 2: train and verify
pnpm ml:train-snapshot --asset BTC --horizon 168 --verify
pnpm ml:train-snapshot --asset ETH --horizon 168 --verify
```

The new ONNX files overwrite the previous ones. The pipeline picks them up on the next brief without restart.

`--verify` re-runs inference via `onnxruntime` immediately after export and confirms parity. Always use it.

### When to retrain

- **Weekly** — more snapshots accumulate; IC improves as sample size grows
- **After regime shifts** — if the market moves from bull to bear, the feature-return correlations may change; retrain to capture the new regime
- **After feature schema changes** — any change to `feature_schema.json` or `extract-features.ts` invalidates existing models; regenerate CSV and retrain

### Experimenting with horizons

The CSV contains four horizons. Try other targets and compare OOF IC:

```bash
pnpm ml:train-snapshot --asset BTC --horizon 72 --verify   # 3-day target
pnpm ml:train-snapshot --asset BTC --horizon 48 --verify   # 2-day target
```

Use the horizon with the highest OOF IC. Currently 168h (1 week) is the default for swing-trading alignment.

## Known limitations

**Sample size**: 2 months of 4H snapshots = ~600 per asset. This is an improvement over the 300 trade ideas before, but still small for 77 features. OOF IC of 0.63 is promising; retrain monthly and watch whether it holds as more data arrives.

**Single market regime**: The training window (March–May 2026) covered a specific market regime. The model may underperform during a different regime (e.g., prolonged sideways chop) until retrained on regime-diverse data.

**Top features are mean-reversion signals**: `structuralPriceVsPocPct`, `mfiH4`, and related features measure how stretched price is from structural levels — these tend to predict the reversion after overshoots, not necessarily the momentum during the trend. This is correct behavior for a 168h swing horizon but may misfire in very strong trending markets where overshoots keep extending.

**Ridge predictions are magnitude-shrunk**: The model predicts direction well (high IC) but the absolute scores are conservative — a score of 0.3 does not mean "30% expected return." Only the sign and relative ranking of scores across time matters for direction + sizing decisions.

## Legacy fallback models (L2a + L1)

The L2a per-dimension models (`dim_*_v1.onnx`) and the L1 cross-dim aggregator (`confluence_*_v1.onnx`) remain on disk as the fallback chain. They run only when the snapshot model file is missing. Their training is described in `train_dim.py` and `train.py` but they are not the primary scoring path.
